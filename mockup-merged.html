Context Engineering for AI Coding Tools

Most teams adopt AI coding tools the same way: install the extension, give it access to the repo, and hope the suggestions are good. When they're not, the conclusion is "AI isn't ready yet" or "it doesn't understand our codebase." Both conclusions are wrong. The tool is fine. The context is broken.

Convention files are the missing layer. A CLAUDE.md, AGENTS.md, or .cursorrules file sitting in your repo root is how you tell the AI tool what your codebase actually cares about: the naming conventions, the architectural decisions, the things that would take a new hire three months to absorb. Without that file, the AI is guessing. With it, the AI starts compounding.

The context problem isn't new

Every senior engineer has watched a junior developer write technically correct code that violates every unwritten rule in the codebase. The function works. The tests pass. But the naming is wrong, the error handling follows a pattern the team abandoned six months ago, and the database query uses an approach that'll fall apart at scale. The code review catches it, but the damage is a wasted cycle.

AI coding tools have the same problem, except they never learn from the code review. Every session starts fresh. Every suggestion is context-free unless you build the context layer yourself.

What a convention file actually does

A convention file is a plain text file in your repo that the AI tool reads at the start of every session. It's not a prompt. It's not a system instruction you paste into a chat window. It's a permanent artifact committed alongside your code, versioned with git, reviewed in PRs, and updated as the codebase evolves.

The best convention files I've built share three properties: they describe what the codebase cares about (not what the AI should do), they're specific enough to be falsifiable (not "write clean code" but "error types live in src/errors/ and extend BaseAppError"), and they're short enough that the AI actually reads the whole thing instead of summarizing and forgetting the details.

I ran Claude Code for two months on the same production repo, one month without a convention file, one month with a detailed CLAUDE.md. The diff wasn't subtle. Code review rejection rate dropped from roughly 40% to under 15%. Not because the model got smarter. Because it stopped guessing about things the team had already decided.
